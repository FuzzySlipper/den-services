# Documents Lifeboat Contract

This is the extraction contract for Den Core documents, document search,
visibility/archive behavior, and document-scoped discussion threads/comments.
It follows the projects, tasks, and messages lifeboat contracts: the documents
service owns document content and document discussions, but it must not become a
new Core-shaped aggregator for knowledge, agent guidance, librarian retrieval,
projects, tasks, or messages.

The successor service is:

- Go package/service: `den-services/documents`
- Postgres schema: `den_documents`
- Runtime role: `den_documents_app`
- Primary tables: `den_documents.documents`,
  `den_documents.discussion_threads`, and
  `den_documents.discussion_comments`
- Project authority: `den-services/projects`, checked through the projects API
  or an approved projects read projection before writes
- Deployment pattern: the lifeboat substrate in
  [`docs/lifeboat-service-substrate.md`](./lifeboat-service-substrate.md)

Do not create a placeholder service only to satisfy a smoke test. Register
`documents` in `deployment/services.yaml` only when the real document/search
and discussion/comment contract is implemented.

## Boundary Statement

`documents` owns durable authored project/space documents and the first-class
discussion threads/comments attached to those documents. Discussion data is
stored beside documents initially because the green path is document-targeted:
`comment_on_document` auto-creates a default document thread and writes one
comment.

Document comments are not document body revisions. `get_document` returns the
canonical document JSON with no embedded discussion fields. Discussion reads
and writes use discussion APIs.

This service does not own:

- `knowledge` entries, curation states, revisions, tags, or guided retrieval;
- `agent_guidance_entries`, though archive preflight must report active
  guidance references through a guidance-owned API or approved projection;
- librarian synthesis, context gathering, or cross-domain ranking;
- project/space lifecycle or archived-scope write policy;
- task messages, notifications, task-thread packets, or message read state.

Knowledge must stay separate. Do not back knowledge entries by documents or
route knowledge tools through `documents`.

Current Core document table:

```sql
create table den_core.documents (
    id bigint generated by default as identity primary key,
    project_id text not null references den_core.projects(id) on delete cascade,
    slug text not null,
    title text not null,
    content text not null,
    doc_type text not null default 'spec'
        check (doc_type in ('prd','spec','adr','convention','reference','note','memory')),
    visibility text not null default 'normal'
        check (visibility in ('normal','hidden','archived')),
    tags jsonb,
    summary text,
    search_vector tsvector generated always as (
        to_tsvector('english',
            coalesce(title,'') || ' ' || coalesce(content,'') || ' ' || coalesce(tags::text,'')
        )
    ) stored,
    created_at timestamptz not null default now(),
    updated_at timestamptz not null default now(),
    unique(project_id, slug)
);
create index idx_documents_project_type on den_core.documents(project_id, doc_type);
create index idx_documents_search_vector on den_core.documents using gin(search_vector);
```

## Current Behavior To Preserve

Document behavior:

- `store_document` creates or updates a document by `(project_id, slug)`.
  Upsert replaces title/content/doc_type/tags/summary and updates
  `updated_at`; visibility remains unchanged on updates.
- `get_document` returns documents regardless of visibility. Concise mode
  returns metadata plus content preview; verbose mode returns full content.
- `list_documents` returns summaries, excluding hidden and archived documents
  by default. Hidden or archived documents require an explicit visibility
  filter.
- `search_documents` searches normal documents only. It supports Postgres
  `websearch_to_tsquery` behavior including phrases and operators, and returns
  snippets/rank.
- `delete_document` deletes a document by `(project_id, slug)`.
- `update_document_visibility` changes visibility to `normal`, `hidden`, or
  `archived`.
- `archive_document_preflight` reports active references that would be broken
  or misleading if the document were archived. Core currently reports
  `agent_guidance_entries` references.
- `query_archived_documents` is the deliberate archived recall path. It can
  list archived documents or search archived documents separately from default
  list/search.

Discussion behavior:

- `get_document_discussion` verifies the document exists. With
  `create_if_missing=false`, it does not create a default thread; if none
  exists, it returns an empty result. With `create_if_missing=true`, it
  idempotently creates/returns the default or anchor-specific thread.
- `comment_on_document` is the green path. It verifies the document exists,
  creates the default or anchor-specific thread if needed, and writes exactly
  one comment or reply.
- `create_discussion_comment` is the low-level thread-ID path. Use it when a
  caller explicitly provides `thread_id`.
- `list_discussion_threads` is document-targeted in this phase. It requires or
  strongly expects `target_type=document`, `target_project_id`, and
  `target_slug`; unfiltered cross-target listing is not a required green path.
- `get_discussion_thread` reads one thread and optionally includes comments in
  chronological order.
- `update_discussion_thread` updates thread status/title/summary/resolution
  fields and is admin-current only in Core.
- Replies must validate that the parent comment exists and belongs to the same
  thread.
- Adding any comment updates `last_comment_at` and `updated_at` on the owning
  thread.

## Inbound Dependencies

Known dependents include:

- agent guidance, which references documents by `document_project_id` and
  `document_slug`;
- librarian/context gathering, which searches and reads documents but does not
  own them;
- MCP profile/bundle routing for document and discussion tools;
- project/space summaries and cleanup flows that need to reason about archived
  documents;
- review/planning workflows that comment on specs or resolve discussion
  threads;
- future import/parity tooling that migrates Core document and discussion rows.

Dependent domains should store `(project_id, slug)` for document references
and `discussion_thread_id` / `discussion_comment_id` where needed. They must
not write `den_documents` tables directly. Cross-domain cascade deletes are
forbidden outside this schema; foreign keys from later services to documents or
discussion rows should use `ON DELETE RESTRICT` or `ON DELETE SET NULL`, not
cascade.

## New Schema Boundary

The owning schema is `den_documents`.

```sql
create table den_documents.documents (
    id bigint generated always as identity primary key,
    project_id text not null,
    slug text not null,
    title text not null,
    content text not null,
    doc_type text not null default 'spec',
    visibility text not null default 'normal',
    tags jsonb,
    summary text,
    search_vector tsvector generated always as (
        setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
        setweight(to_tsvector('english', coalesce(summary, '')), 'B') ||
        setweight(to_tsvector('english', coalesce(content, '')), 'C') ||
        setweight(to_tsvector('english', coalesce(tags::text, '')), 'D')
    ) stored,
    created_at timestamptz not null default now(),
    updated_at timestamptz not null default now(),
    unique(project_id, slug)
);

create table den_documents.discussion_threads (
    id bigint generated always as identity primary key,
    target_type text not null default 'document',
    target_project_id text not null,
    target_id bigint,
    target_slug text,
    target_anchor text,
    thread_key text not null,
    title text not null,
    status text not null default 'open',
    created_by text not null,
    summary text,
    resolution_summary text,
    metadata_json jsonb,
    last_comment_at timestamptz,
    created_at timestamptz not null default now(),
    updated_at timestamptz not null default now()
);

create table den_documents.discussion_comments (
    id bigint generated always as identity primary key,
    thread_id bigint not null references den_documents.discussion_threads(id) on delete cascade,
    parent_comment_id bigint references den_documents.discussion_comments(id) on delete cascade,
    author_identity text not null,
    body_markdown text not null,
    comment_kind text not null default 'comment',
    status text not null default 'active',
    mentions_json jsonb,
    source_refs_json jsonb,
    metadata_json jsonb,
    created_at timestamptz not null default now(),
    edited_at timestamptz,
    updated_at timestamptz not null default now()
);
```

Recommended indexes:

- `den_documents.documents(project_id, slug)` unique
- `den_documents.documents(project_id, doc_type, updated_at desc, id desc)`
- `den_documents.documents(project_id, visibility, updated_at desc, id desc)`
- `den_documents.documents(visibility, updated_at desc, id desc)`
- `den_documents.documents using gin(search_vector)`
- `den_documents.documents using gin(tags jsonb_path_ops)`
- `den_documents.discussion_threads(target_project_id, target_slug, status, updated_at desc, id desc)`
- unique discussion target key over `target_type`, `target_project_id`,
  `target_slug`, `target_id`, `target_anchor`, and `thread_key`, preserving
  Core's `coalesce` semantics
- `den_documents.discussion_comments(thread_id, created_at asc, id asc)`
- `den_documents.discussion_comments(parent_comment_id)` where
  `parent_comment_id is not null`

Service rules:

- `project_id` / `target_project_id` are scope references owned by the projects
  service. The documents service validates project existence and archived-scope
  write policy through `projects` before document or discussion writes.
- `slug`, `title`, `content`, `body_markdown`, and `author_identity` are
  required and trimmed at the service boundary.
- Document tags are JSON arrays of strings. Tag filters use AND semantics and
  must not match substrings.
- `store_document` never creates knowledge entries.
- Discussion target type is `document` only in this phase. Other target types
  should be rejected, not silently stored.
- Default thread key is `default`. Anchor-specific thread key is
  `section:{trimmed_anchor}`.
- `comment_on_document` must auto-create the default or anchor-specific thread
  idempotently.
- `get_document_discussion` must not create a thread unless
  `create_if_missing=true`.
- Root comments have `parent_comment_id=null`; replies must reference a comment
  in the same thread.
- Thread/comment JSON fields (`metadata_json`, `mentions_json`,
  `source_refs_json`) are opaque payloads owned by the caller, but the service
  must validate that supplied values are valid JSON.

## Postgres Search Contract

The successor uses native Postgres full-text search, not SQLite FTS and not
application-side substring search.

Search vector:

- title: weight `A`
- summary: weight `B`
- content: weight `C`
- tags text: weight `D`

Query execution:

- Reject empty/whitespace queries before hitting the database.
- Use `websearch_to_tsquery('english', query)` so phrase/operator behavior
  remains close to Core.
- Use `ts_rank_cd(search_vector, query)` for ranking.
- Use `ts_headline('english', coalesce(content, summary, title, ''), query,
  headline_options)` for snippets.
- Preserve Core headline options unless review decides otherwise:
  `StartSel=<b>, StopSel=</b>, MaxWords=32, MinWords=8, ShortWord=3,
  HighlightAll=false, MaxFragments=2, FragmentDelimiter=...`.
- `search_documents` filters `visibility='normal'`.
- Archived search filters `visibility='archived'` and is available only through
  the deliberate archived recall route/tool.
- Hidden documents are not returned by default list/search. `get_document` may
  fetch them by exact `(project_id, slug)`.

## Import And Cutover Notes

Migration/import should copy:

- `den_core.documents`
- `den_core.discussion_threads` where `target_type='document'`
- `den_core.discussion_comments` for copied threads

Import must preserve IDs only if the implementation explicitly supports
identity insert and downstream references need stable IDs. Otherwise, maintain
an import map from old IDs to new IDs for discussion threads/comments.

Cutover order:

1. Create schema/tables/roles/grants.
2. Import documents and discussions into `den_documents`.
3. Run parity checks for counts, `(project_id, slug)` uniqueness, visibility
   filters, archived recall, FTS search, default-thread lookup, comment order,
   parent reply integrity, and thread status updates.
4. Route read-only document tools first.
5. Route document write tools after store/update/delete/archive smokes.
6. Route discussion/comment write tools after default-thread and reply smokes.
7. Keep knowledge/guidance/librarian routes unchanged.

## Required Smokes

Document smokes:

1. Store a document, get it by exact `(project_id, slug)`, then upsert it and
   verify content/title/summary/tags changed while visibility is preserved.
2. List documents and verify hidden/archived documents are excluded by default.
3. List with `visibility=hidden` and `visibility=archived` and verify explicit
   inclusion.
4. Search normal documents and verify archived documents do not appear.
5. Query archived documents by list and by search and verify only archived rows
   appear.
6. Update visibility `normal -> archived -> normal`.
7. Archive preflight returns active `agent_guidance` references for a referenced
   document and no references for an unreferenced document.
8. Delete a document and verify `get_document` returns not found.

Discussion/comment smokes:

1. `get_document_discussion` on a document with no threads returns empty and
   creates no row.
2. `get_document_discussion(create_if_missing=true)` creates/returns the default
   thread idempotently.
3. `comment_on_document` auto-creates the default thread and writes one
   root-level comment.
4. `comment_on_document` with `parent_comment_id` writes a reply and rejects a
   parent from another thread.
5. Anchor comments use `thread_key='section:{anchor}'` and are separate from the
   default thread.
6. `create_discussion_comment` writes to an explicit thread ID and rejects
   missing threads.
7. `get_discussion_thread(include_comments=true)` returns comments oldest-first.
8. `update_discussion_thread` can mark a thread `resolved` with
   `resolution_summary`, then `archived`.

Gateway/MCP smokes:

1. MCP `store_document`, `get_document`, `list_documents`, and
   `search_documents` route to the successor and preserve concise/verbose
   behavior.
2. MCP `query_archived_documents` is the only archived search/list route used
   by default callers.
3. MCP `comment_on_document` succeeds without a prior `get_document_discussion`.
4. MCP `get_document_discussion` with `create_if_missing=false` remains
   read-only.
5. MCP knowledge and agent-guidance tools remain on their own owners.

## Acceptance For Implementation Task

An implementation task for this contract should include:

- `den_documents` migrations and grants;
- a real `documents` Go service following Handler -> Service -> Store layering;
- focused unit tests for validation, visibility filters, FTS query handling,
  default thread creation, replies, and status updates;
- Postgres-backed tests or transactional smokes for FTS, archive preflight,
  unique thread keys, and comment parent integrity;
- MCP route mapping documentation that explicitly leaves knowledge, guidance,
  and librarian off this service;
- `go test ./...` evidence for the new module and the workspace loop.
