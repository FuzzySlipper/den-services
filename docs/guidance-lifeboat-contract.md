# Guidance Lifeboat Contract

This is the extraction contract for Den agent guidance entries, resolved
guidance packets, and tool-document lookup used by fail-closed packet
validation. It follows the projects and documents lifeboat contracts: guidance
owns policy/reference assembly, but it does not own project lifecycle, document
content, knowledge retrieval, librarian synthesis, messages, tasks, review, or
worker orchestration.

The successor service is:

- Go package/service: `den-services/guidance`
- Postgres schema: `den_guidance`
- Runtime role: `den_guidance_app`
- Primary tables: `den_guidance.agent_guidance_entries` and
  `den_guidance.tool_docs`
- Project authority: `den-services/projects`, checked through the projects API
  or an approved projects read projection before writes
- Document authority: `den-services/documents`, checked through document APIs
  or approved document read projections before guidance entries are created or
  resolved
- Deployment pattern: the lifeboat substrate in
  [`docs/lifeboat-service-substrate.md`](./lifeboat-service-substrate.md)

Do not create a placeholder service only to satisfy a smoke test. Register
`guidance` in `deployment/services.yaml` only when the real guidance contract is
implemented.

## Boundary Statement

`guidance` owns first-class agent guidance entries and deterministic resolved
guidance packets. A guidance entry is a policy reference from a scope to an
existing document. The document body remains owned by `documents`; guidance
stores only the reference plus ordering and audience metadata.

`guidance` also owns compact tool documentation lookup for validation and
recovery paths such as:

```text
tool_docs("post_review_packet_markdown", "review_findings.verdict")
```

Tool docs are operational reference material, not project policy. They must not
be blindly appended to resolved agent guidance packets. Packet validators and
MCP tools may return tool-doc handles when they reject malformed input, and
callers can ask guidance for the small reference card attached to that handle.

This service does not own:

- project/space lifecycle, archived-scope write policy, or project metadata;
- authored document content, document search, document discussions, or document
  visibility changes;
- knowledge entries, curation, revisions, guided retrieval, or librarian
  synthesis;
- AGENTS snapshots, local workspace bootstrap files, or repo-specific
  instruction discovery;
- task messages, notifications, packet storage, review workflow, worker runs,
  capabilities, or orchestrator state.

Guidance is not a new Core-shaped context aggregator. Resolved guidance packets
must remain deterministic, bounded, source-attributed, and explicit about
omitted or unavailable sources.

## Existing Core Inventory

Core guidance ownership is currently centered in:

- `AgentGuidanceRepository`
- `AgentGuidanceEntry`
- `ResolvedAgentGuidance`
- `AgentGuidanceRoutes`
- `AgentGuidanceTools`

Current Core table:

```sql
create table den_core.agent_guidance_entries (
    id bigint generated by default as identity primary key,
    project_id text not null references den_core.projects(id) on delete cascade,
    document_project_id text not null references den_core.projects(id) on delete cascade,
    document_slug text not null,
    importance text not null default 'important'
        check (importance in ('required','important')),
    audience text,
    sort_order integer not null default 0,
    notes text,
    created_at timestamptz not null default now(),
    updated_at timestamptz not null default now(),
    unique (project_id, document_project_id, document_slug)
);

create index idx_agent_guidance_scope_order
    on den_core.agent_guidance_entries(project_id, importance, sort_order, id);

create index idx_agent_guidance_document
    on den_core.agent_guidance_entries(document_project_id, document_slug);
```

Current REST behavior to preserve:

- `GET /api/projects/{projectId}/agent-guidance` resolves a project guidance
  packet.
- `GET /api/projects/{projectId}/agent-guidance/entries?includeGlobal=...`
  lists configured entries.
- `POST /api/projects/{projectId}/agent-guidance/entries` upserts one
  document-linked entry after verifying that the target document exists.
- `DELETE /api/projects/{projectId}/agent-guidance/entries/{entryId}` deletes
  one entry scoped to the route project.

Current MCP tools:

- `get_agent_guidance`
- `list_agent_guidance_entries`
- `add_agent_guidance_entry`
- `delete_agent_guidance_entry`

Current entry fields:

- `id`
- `project_id`
- `document_project_id`
- `document_slug`
- `importance`, with values `required` or `important`
- `audience`, currently comma-separated text at the MCP boundary
- `sort_order`
- `notes`
- `created_at`
- `updated_at`

Current list and resolve ordering is deterministic:

1. global scope before project scope;
2. `sort_order` ascending;
3. `required` before `important`;
4. `document_project_id` ascending;
5. `document_slug` ascending;
6. `id` ascending.

Current scope behavior:

- Resolving project `_global` includes only `_global` entries.
- Resolving any other project includes `_global` entries and that project's
  entries.
- Listing entries includes `_global` entries only when the caller passes
  `includeGlobal=true`.

Current resolved content behavior:

- A packet starts with `# Den Agent Guidance for {projectId}`.
- A packet contains an included-sources section and guidance-content section.
- Each source shows importance, scope, document reference, document type,
  sort order, and audience.
- Each guidance section includes the referenced document title, metadata,
  tags, audience, notes, and full document content.
- Core excludes archived documents while resolving guidance. Hidden documents
  currently still resolve; the successor must tighten this boundary as
  described below.

Current document archive preflight behavior:

- `archive_document_preflight` reports active `agent_guidance_entries`
  references before a document is archived.
- After extraction, this must be served by guidance through an API or approved
  projection; `documents` must not own or write guidance rows.

## Ownership Decision

Guidance is its own service, not a submodule inside `projects`.

The project/scope service owns whether a scope exists and whether writes are
allowed to that scope. Guidance owns the policy references attached to a scope,
their deterministic merge order, and the resolved packet shape. Keeping this as
a separate service prevents projects from becoming a content aggregation
service and keeps document-linked policy references out of the projects schema.

Tool-doc lookup also belongs in `guidance`, but as a separate operational
reference subdomain. Tool docs are small, keyed reference cards used when a
validator or MCP facade needs to explain an error or enum. They are not scoped
policy and they are not part of default resolved agent guidance.

## New Schema Boundary

The owning schema is `den_guidance`.

```sql
create table den_guidance.agent_guidance_entries (
    id bigint generated by default as identity primary key,
    project_id text not null,
    document_project_id text not null,
    document_slug text not null,
    importance text not null default 'important',
    audience jsonb,
    sort_order integer not null default 0,
    notes text,
    created_at timestamptz not null default now(),
    updated_at timestamptz not null default now(),
    unique (project_id, document_project_id, document_slug)
);

create table den_guidance.tool_docs (
    tool_name text not null,
    topic text not null,
    title text not null,
    summary text,
    content_markdown text not null,
    allowed_values jsonb,
    examples jsonb,
    status text not null default 'active',
    created_at timestamptz not null default now(),
    updated_at timestamptz not null default now(),
    primary key (tool_name, topic)
);

create table den_guidance.tool_doc_aliases (
    tool_name text not null,
    alias text not null,
    topic text not null,
    created_at timestamptz not null default now(),
    primary key (tool_name, alias),
    foreign key (tool_name, topic)
        references den_guidance.tool_docs(tool_name, topic)
        on delete cascade
);
```

Recommended indexes:

- `den_guidance.agent_guidance_entries(project_id, sort_order, importance,
  document_project_id, document_slug, id)`
- `den_guidance.agent_guidance_entries(document_project_id, document_slug)`
- `den_guidance.tool_docs(tool_name, status)`

Service rules:

- `project_id` is a scope reference owned by `projects`. It may be `_global` or
  a project/space ID that projects recognizes.
- `document_project_id` and `document_slug` identify a document owned by
  `documents`.
- The database must not use cross-schema cascade foreign keys into projects or
  documents. Guidance validates references through service APIs or approved
  projections and exposes reference checks back to documents.
- `importance` values are `required` and `important`. Validate in Go rather
  than a database `CHECK` constraint.
- `audience` is a JSON array of strings in storage. MCP compatibility may keep
  accepting comma-separated text and convert it at the boundary.
- `sort_order` is an integer. Lower values resolve earlier.
- `tool_docs.status` values are `active`, `hidden`, and `deprecated`. Validate
  in Go.
- `tool_doc_aliases` allows packet validators to return stable error codes
  while documentation topics are organized by semantic topic.

## Guidance Entry Semantics

Create or update:

- `POST` upserts by `(project_id, document_project_id, document_slug)`.
- `document_project_id` defaults to the route `project_id` unless supplied.
- The service must reject missing projects/scopes.
- The service must reject missing documents.
- The service must reject archived documents.
- The service must reject hidden documents unless the caller uses an explicit
  admin-only override such as `allow_hidden_document=true`.
- The service must reject invalid `importance`, malformed `audience`, and
  non-string audience items.

List:

- `include_global=false` lists only entries owned by the requested scope.
- `include_global=true` lists `_global` first and the requested scope second.
- Listing returns entry metadata and document references. It does not need to
  inline full document bodies.

Delete:

- Delete is by entry ID plus owning `project_id`.
- Deleting a missing entry is a not-found result, not a silent success.
- Deleting an entry must remove it from subsequent resolve results.

Archive and visibility:

- `documents` calls guidance before archiving or hiding a referenced document.
- Guidance returns active references by `(document_project_id, document_slug)`.
- Archive preflight should block or report references even when the guidance
  entry is project-scoped and the document is global-scoped, or vice versa.
- Resolved packets never include archived document content.
- Resolved packets never include hidden document content unless the entry was
  created with an explicit hidden-document override and the caller is allowed to
  see hidden guidance.

## Resolve Semantics

Resolve returns a packet, not just concatenated Markdown.

The packet fields are:

- `project_id`
- `resolved_at`
- `sources`
- `skipped_sources`
- `content_markdown`
- `content_sha256`
- `content_bytes`
- `truncated`
- `incomplete`

`sources` includes one item per document included in `content_markdown`:

- guidance entry ID;
- source scope;
- document project ID;
- document slug;
- document title;
- document type;
- document updated timestamp when available;
- visibility used for the decision;
- importance;
- audience;
- sort order;
- notes;
- content byte count or digest when available.

`skipped_sources` includes entries that were configured but not included:

- missing document;
- archived document;
- hidden document without override or visibility authority;
- document service unavailable;
- packet byte budget exceeded.

`incomplete=true` whenever any configured source is skipped for reasons other
than audience filtering. Required skipped sources must be surfaced with a
machine-readable diagnostic such as `required_guidance_source_unavailable`.

Default ordering must preserve Core behavior:

1. global scope before project scope;
2. `sort_order` ascending;
3. `required` before `important`;
4. `document_project_id` ascending;
5. `document_slug` ascending;
6. `id` ascending.

Audience filtering is optional at first cutover. If implemented, it must be an
explicit request parameter and must not change default Core-compatible resolve
behavior.

Resolved guidance must be bounded. The service should enforce a configurable
maximum packet byte budget and return `truncated=true` or a
`guidance_packet_too_large` diagnostic instead of silently returning an
unbounded context dump. MCP compatibility may continue returning Markdown, but
the successor API must preserve source metadata so clients can recover by
fetching specific documents.

## Tool Docs Semantics

Tool docs are compact reference cards keyed by `(tool_name, topic)`.

Lookup behavior:

- `GET` by tool name and topic returns only active docs by default.
- Alias lookup maps `(tool_name, alias)` to a canonical topic.
- Missing docs return a typed not-found response that includes the requested
  handle.
- Hidden docs require an explicit admin or diagnostic mode.
- Deprecated docs may be returned only when requested explicitly, with status
  metadata intact.

Tool doc response fields:

- `tool_name`
- `topic`
- `title`
- `summary`
- `content_markdown`
- `allowed_values`
- `examples`
- `status`
- `updated_at`

Tool docs must stay small enough to be useful at error time. They should explain
the rejected field, valid values, and one or two repair examples. They should
not embed full tool manuals or project guidance.

Packet validators and MCP facades should return documentation handles instead
of prose-only errors. Example diagnostic:

```json
{
  "code": "invalid_verdict",
  "field": "verdict",
  "message": "review_findings.verdict must be one of changes_requested, looks_good, follow_up_needed, blocked_by_dependency",
  "docs_ref": {
    "kind": "tool_docs",
    "tool_name": "post_review_packet_markdown",
    "topic": "review_findings.verdict"
  }
}
```

## Service API

Native guidance API:

- `GET /v1/projects/{project_id}/agent-guidance`
- `GET /v1/projects/{project_id}/agent-guidance/entries?include_global=false`
- `POST /v1/projects/{project_id}/agent-guidance/entries`
- `DELETE /v1/projects/{project_id}/agent-guidance/entries/{entry_id}`
- `GET /v1/guidance/document-references?document_project_id=...&document_slug=...`
- `GET /v1/tool-docs/{tool_name}/{topic}`
- `GET /v1/tool-docs/{tool_name}/aliases/{alias}`

The resolved guidance route supports:

- `include_content=true|false`
- `max_bytes=...`
- `audience=...`
- `include_hidden=false|true` for explicitly authorized diagnostics only

The document references route returns enough data for document archive
preflight:

- guidance entry ID;
- owning guidance scope;
- document project ID;
- document slug;
- importance;
- audience;
- sort order;
- notes;
- created and updated timestamps.

Compatibility routes may keep the Core `/api/projects/{projectId}` shape while
the MCP facade and desktop clients migrate.

## MCP Mapping

Existing MCP tools should preserve names while moving backing reads/writes to
`guidance`:

- `get_agent_guidance(project_id)`
- `list_agent_guidance_entries(project_id, include_global=false)`
- `add_agent_guidance_entry(project_id, document_slug, document_project_id,
  importance, audience, sort_order, notes, verbose=false)`
- `delete_agent_guidance_entry(project_id, entry_id)`

New MCP lookup:

- `tool_docs(tool_name, topic_or_error_code)`

MCP compatibility notes:

- `get_agent_guidance` may keep returning concise Markdown by default, but the
  service contract is packet-shaped and source-attributed.
- `list_agent_guidance_entries` should not inline document content.
- `add_agent_guidance_entry` should continue accepting comma-separated
  `audience`, converting to JSON array storage.
- `tool_docs` returns a compact reference card and must not resolve project
  guidance or query librarian.

## Cross-Service Integration

Projects:

- Guidance asks projects whether a scope exists before creating project-scoped
  entries.
- Guidance asks projects whether writes are allowed before mutating entries.
- Guidance treats `_global` as a reserved scope and requires appropriate write
  authority for global entries.

Documents:

- Guidance asks documents to validate target documents before create/update.
- Guidance asks documents for document metadata and content during resolve.
- Documents ask guidance for active references during archive preflight and
  hidden/archive policy checks.
- Documents do not write `den_guidance` rows.

Packets and review:

- Packet validators return tool-doc handles for field-specific errors.
- Review/task/message services do not own tool docs.
- Guidance does not store packet bodies, review findings, or task messages.

Knowledge and librarian:

- Knowledge remains a separate domain.
- Librarian retrieval may use guidance packet sources as one signal later, but
  guidance does not call librarian for normal resolve.

## Migration And Cutover

Cutover steps:

1. Add the `guidance` service only when real storage and API behavior are ready.
2. Create `den_guidance` schema and tables.
3. Backfill `agent_guidance_entries` from Core with identical IDs when possible.
4. Seed required tool docs for packet validation errors that already return
   structured docs handles.
5. Run parity reads comparing Core and successor list/resolve output for normal
   documents.
6. Switch MCP tools and compatibility REST routes to successor backing.
7. Update document archive preflight to ask guidance for references.
8. Remove Core writes after parity and read-through are stable.

Backfill rules:

- Preserve `id`, `project_id`, `document_project_id`, `document_slug`,
  `importance`, `sort_order`, `notes`, `created_at`, and `updated_at`.
- Convert comma-separated `audience` text to JSON array storage.
- Flag entries pointing at missing, hidden, or archived documents for operator
  review rather than silently dropping them.
- Do not seed tool docs from full project guidance documents.

Rollback:

- Keep Core as the source of truth until MCP tools have switched and parity
  smokes pass.
- If successor resolve fails during cutover, MCP compatibility can read Core
  while writes are paused.
- If tool-doc lookup fails, validators should still return the docs handle and
  typed error; they should not invent uncited long-form help text.

## Smoke Tests

Required smokes before cutover:

- doc-linked guidance: create a normal document, add guidance entry referencing
  it, list entries, and resolve a packet containing the source metadata and
  document content;
- global plus project resolve: add `_global` and project-scoped entries with
  mixed `sort_order` and `importance`, then verify deterministic ordering;
- delete: delete one entry by ID and verify it disappears from list and resolve;
- invalid document: adding an entry for a missing document fails closed;
- archived document: archived referenced documents are not included in resolved
  content and are surfaced in `skipped_sources`;
- hidden document: hidden referenced documents are not included unless an
  explicit authorized hidden-document path is used;
- archive preflight: documents asks guidance for references and receives active
  entries for the document being archived;
- tool docs lookup: a validator returns a docs handle for an invalid packet
  field and `tool_docs` resolves it to a compact reference card;
- tool docs missing handle: missing docs return a typed not-found response with
  the original handle;
- packet budget: an oversized guidance set returns truncation metadata or a
  typed size diagnostic instead of an unbounded response.

Recommended haunt-regression cases:

- resolving `_global` returns only `_global` entries;
- resolving a normal project includes `_global` before project entries;
- `include_global=false` list excludes global entries;
- duplicate upsert changes metadata without creating a second entry;
- audience parsing rejects empty or non-string JSON items;
- hidden/archived source bodies never leak through compatibility MCP output.

## Non-Goals

- Do not migrate knowledge entries into documents or guidance.
- Do not introduce an AGENTS snapshot table in guidance.
- Do not build a fake service skeleton just to pass a deployment smoke.
- Do not let resolved guidance become an unbounded context-dump endpoint.
- Do not route librarian, task context packets, review summaries, or worker
  prompts through guidance unless a later contract explicitly assigns that
  responsibility.
