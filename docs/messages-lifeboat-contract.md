# Messages Lifeboat Contract

This is the extraction contract for Den Core project/task messages, user
notifications, read state, and task-thread packet storage/readback. It follows
the projects and tasks lifeboat contracts: the messages service owns durable
threaded records and read-state projections, but it must not become a new Core
aggregator for tasks, review workflow, worker state, agent stream, or delivery
actuation.

The successor service is:

- Go package/service: `den-services/messages`
- Postgres schema: `den_messages`
- Runtime role: `den_messages_app`
- Primary tables: `den_messages.messages`, `den_messages.message_reads`
- Project authority: `den-services/projects`, checked through the projects API
  or an approved projects read projection before writes
- Task authority: `den-services/tasks`, read through API or approved task
  projection for task-scoped validation and packet context
- Deployment pattern: the lifeboat substrate in
  [`docs/lifeboat-service-substrate.md`](./lifeboat-service-substrate.md)

Do not create a placeholder service only to satisfy a smoke test. Register
`messages` in `deployment/services.yaml` only when the real message/packet
contract is implemented.

## Boundary Statement

`messages` is a durable conversation/readback authority for project and task
threads. Rows in this service are display/instruction records. They can be
replayed for humans and agents to read, but replaying them must not create,
claim, wake, retry, complete, or cancel executable work.

This service is not the same domain as `conversation/` channel chat. The
existing `conversation` module owns channel messages, channel membership,
reactions, and human channel cursors. The messages lifeboat owns Core-style
project/task message threads and user-notification feed items, including
task-thread worker packets.

Worker packets are stored as messages because they are durable task-thread
records. Worker launch, worker run state, assignment state, no-capacity
diagnostics, and orchestration gates remain worker/runtime/orchestrator
domains. `messages` may expose packet lookup and packet append APIs, but it
must not mutate worker tables or decide whether a worker assignment is complete.

## Existing Core Inventory

Core ownership is currently centered in:

- `MessageRepository`
- `Message`
- `MessageRoutes`
- `MessageTools`
- `PacketTools`
- `CompletionTools`, for completion packet message creation plus worker-state
  side effects
- review and task services that create typed packet messages through
  `IMessageRepository`

Current Core tables:

```sql
create table den_core.messages (
    id bigint generated by default as identity primary key,
    project_id text references den_core.projects(id) on delete set null,
    task_id bigint references den_core.tasks(id) on delete set null,
    thread_id bigint references den_core.messages(id) on delete set null,
    sender text not null,
    content text not null,
    intent text not null default 'general',
    metadata jsonb,
    created_at timestamptz not null default now()
);

create index idx_messages_project_task on den_core.messages(project_id, task_id);
create index idx_messages_thread on den_core.messages(thread_id);

create table den_core.message_reads (
    message_id bigint not null references den_core.messages(id) on delete cascade,
    agent text not null,
    read_at timestamptz not null default now(),
    primary key (message_id, agent)
);
```

Current canonical message intents are:

- `general`
- `note`
- `status_update`
- `question`
- `answer`
- `handoff`
- `review_request`
- `review_feedback`
- `review_approval`
- `task_ready`
- `task_blocked`
- `notification`

Core currently validates those values in code and in the legacy DDL. The
successor should validate intents in Go rather than freezing them as a database
`CHECK` constraint because packet/message vocabularies evolve frequently.

Legacy metadata compatibility currently derives canonical intent from
`metadata.type` for known values such as `review_request_packet`,
`rereview_packet`, `review_findings_packet`, `merge_request`, and
`planning_summary`. The successor must preserve this compatibility at the
write boundary during migration.

Current REST behavior to preserve:

- `POST /api/projects/{projectId}/messages/` creates a project/task message or
  reply.
- `GET /api/projects/{projectId}/messages/` lists newest messages with filters
  for task, since, unread-for, limit, and intent.
- `GET /api/projects/{projectId}/messages/{messageId}` reads one message and
  rejects project mismatches.
- `GET /api/projects/{projectId}/messages/thread/{threadId}` returns root plus
  replies oldest-first.
- `GET /api/projects/{projectId}/messages/wait` waits up to a bounded timeout
  for unread messages without callers tight-polling.
- `GET /api/projects/{projectId}/messages/feed` returns threaded feed roots and
  latest replies.
- `POST /api/messages/mark-read` records read state.
- `GET /api/user-notifications` and
  `GET /api/projects/{projectId}/user-notifications/` list notification
  messages across or within projects.
- `POST /api/user-notifications/mark-read` marks notification messages read by
  explicit IDs or scoped mark-all.

## Inbound Dependencies

Known dependents include:

- tasks detail and workflow summaries, which need recent message headers;
- review workflow, which creates review request, re-review, findings, verdict,
  and handoff packets;
- worker/orchestrator services, which need context packet creation, completion
  packet storage/readback, and prompt references;
- blocked-task escalation and stale-attention routing, which create user
  notifications and planner-facing messages;
- projects/spaces summary compatibility, which needs unread-message counts;
- agent stream/routing, which may create wake or attention entries from message
  metadata but must remain a separate domain;
- librarian/context builders that include task-thread messages;
- Den Desktop and Den Web notification feeds.

Dependent domains should store `message_id bigint`, `thread_id bigint`,
`project_id text`, and `task_id bigint` as references when needed. They must
not write `den_messages` tables directly. Cross-domain cascade deletes are
forbidden outside the messages schema; foreign keys from later services to
messages should use `ON DELETE RESTRICT` or `ON DELETE SET NULL`.

## New Schema Boundary

The owning schema is `den_messages`.

```sql
create table den_messages.messages (
    id bigint generated by default as identity primary key,
    project_id text not null,
    task_id bigint,
    thread_id bigint references den_messages.messages(id) on delete set null,
    sender text not null,
    content text not null,
    intent text not null default 'general',
    metadata jsonb,
    created_at timestamptz not null default now()
);

create table den_messages.message_reads (
    message_id bigint not null references den_messages.messages(id) on delete cascade,
    agent text not null,
    read_at timestamptz not null default now(),
    primary key (message_id, agent)
);
```

Recommended indexes:

- `den_messages.messages(project_id, task_id, created_at desc, id desc)`
- `den_messages.messages(project_id, created_at desc, id desc)`
- `den_messages.messages(thread_id, created_at, id)`
- `den_messages.messages(project_id, intent, created_at desc, id desc)`
- `den_messages.messages((metadata->>'type'))` where `metadata ? 'type'`
- `den_messages.messages((metadata->>'packet_kind'))` where
  `metadata ? 'packet_kind'`
- `den_messages.messages((metadata->>'run_id'))` where `metadata ? 'run_id'`
- `den_messages.message_reads(agent, message_id)`

Service rules:

- `project_id` is required. The messages service validates project existence
  and archived-project write policy through `projects` before message or
  notification writes.
- `task_id` is optional. If present, the service validates the task exists and
  belongs to `project_id` through `tasks` before writes.
- `thread_id`, when present, must reference an existing root message in the
  same project, and if `task_id` is present it must match the root task.
- Thread replies store `thread_id` pointing at the root message ID. A root
  message has `thread_id is null`; callers may treat `coalesce(thread_id, id)`
  as the thread root.
- Messages are append-only. Do not add edit/delete semantics in this lifeboat
  unless a later task explicitly defines moderation or redaction.
- `message_reads` is idempotent per `(message_id, agent)`.
- `intent` values are validated in Go. Do not use a DDL `CHECK` for the
  intent vocabulary.
- `metadata` is allowed because packet and routing metadata is an extensible
  envelope, but known packet schemas must be validated by typed request
  handlers before storage.
- Secrets must never be stored in `content` or metadata. Packet builders must
  preserve existing redaction guidance.

## Notifications

User notifications are not a separate table in the first lifeboat. They are
messages with `intent='notification'` and metadata carrying notification
facets.

Required metadata keys:

- `urgency`: `low`, `normal`, or `high`; default `normal`
- `type`: optional subtype such as `agent_work_complete`,
  `scope_audit_planner_route`, or `blocked_task_escalation`

Notification feed filters:

- `project_id`
- `task_id`
- `sender`
- `metadata_type`
- `urgency`
- `is_read`, which requires `read_for_agent`
- `limit` clamped to `1..100`
- `offset` clamped to `>=0`

`mark_notifications_read` has two modes:

1. Explicit IDs: mark only supplied IDs that are notification messages.
2. Scoped mark-all: mark all notification messages in a project, optionally
   narrowed to one task.

The modes are mutually exclusive. Requests without either mode are rejected.

## Wait Semantics

`wait_for_messages` preserves Core's bounded wait contract:

- Requires `project_id` and `unread_for`.
- Excludes messages already read by `unread_for`.
- Excludes messages sent by `unread_for`.
- Supports an optional cursor that skips message IDs less than or equal to the
  cursor.
- Clamps timeout to `500..60000` ms.
- Clamps limit to `1..100`.
- Checks for work every 500 ms or less until timeout.
- Returns compact headers only: ID, sender, task ID, content preview, and
  created time.
- On timeout, returns a receipt that tells the caller to stop polling and wait
  for external wake/human input.

The implementation may use Postgres `LISTEN/NOTIFY` later, but the initial
contract should not require it. Polling inside the service is acceptable
because it is bounded, sleeps between checks, and centralizes the wait behavior
that clients must not reproduce.

## Task-Thread Packets

Context packets are messages with typed metadata and `intent='handoff'`.

Required metadata for worker context packets:

- `type`
- `packet_kind`
- `schema='den_worker_packet'`
- `schema_version=1`
- `role`
- `task_id`
- `reference_only_launch=true`
- `completion_reporting_mode`

Supported context packet types:

- `coder_context_packet`
- `reviewer_context_packet`
- `validator_context_packet`
- `drift_checker_context_packet`
- `packet_auditor_context_packet`
- `scope_auditor_context_packet`

Packet creation composes context from:

- task metadata, description, dependencies, and status from `tasks`;
- recent task-thread message headers from `messages`;
- review summaries/findings from the later `review` service when available;
- branch/base/head/scope guidance supplied by the orchestrator.

Until review is extracted, packet builders that require review state may remain
Core-routed or MCP-composed. The messages service may own storage and lookup
for packet messages without owning all packet content builders on day one.

`get_latest_task_packet` is a messages-owned read once packet messages have
been imported:

- Query by `project_id`, `task_id`, optional `packet_type`, and optional
  normalized role.
- Search newest task messages first.
- Match messages whose metadata `type` or `packet_kind` names a packet.
- Return the exact message reference needed by worker launch.

`render_worker_prompt` is a messages-owned read once packet messages have been
imported:

- Validate the packet message exists in `project_id`.
- Return a small prompt that references `project_id`, `task_id`, message ID,
  packet type, role, and completion reporting mode.
- Do not embed the full packet body into process args.

## Completion Packets

Completion packets are durable messages, but posting them has worker-domain
side effects today: Core validates the worker run/assignment and transitions
assignment state after storing the completion packet.

Split the authority as follows:

- `messages` owns storing and reading completion packet messages.
- The future worker/orchestrator service owns run lookup, assignment
  validation, role binding, and assignment state transitions.
- MCP `post_worker_completion_packet` should remain Core-routed or
  worker-service-routed until that worker authority exists, unless MCP composes
  a two-step call that first validates/transitions with the worker service and
  then appends a completion packet through messages.

Required completion metadata when stored in messages:

- `type`
- `packet_kind`
- `schema='den_worker_completion'`
- `schema_version=1`
- `completion_packet=true`
- `status`
- `role`
- `project_id`
- `task_id`
- `run_id`
- optional `branch`, `head_commit`, `base_commit`, `tests_run`,
  `review_round_id`, `finding_ids`, `failure_category`, `recovery_guidance`,
  `dedupe_key`, and scope-audit fields

`get_latest_worker_completion` may become messages-owned when it only needs to
read completion packet messages by `project_id`, optional `task_id`,
`run_id/session_id`, and role. It must return `missing_packet` diagnostics
without mutating worker state.

`post_worker_completion_packet` is not messages-owned as a single-authority
operation until the worker replacement is present.

## Service API

Native routes should live under `/v1`.

Message routes:

- `POST /v1/projects/{project_id}/messages`
  - Request: `sender`, `content`, optional `task_id`, `thread_id`, `intent`,
    `metadata`.
  - Validates project scope and optional task/thread relationships.
- `GET /v1/projects/{project_id}/messages`
  - Filters: `task_id`, `since`, `unread_for`, `limit`, `intent`.
  - Returns newest first.
- `GET /v1/projects/{project_id}/messages/{message_id}`
  - Rejects project mismatches.
- `GET /v1/projects/{project_id}/messages/feed`
  - Returns thread roots, latest message, reply count, and latest activity.
- `GET /v1/messages/{message_id}`
  - Optional direct message read for internal service composition.

Thread routes:

- `GET /v1/projects/{project_id}/messages/threads/{thread_id}`
  - Returns root and replies.
- `GET /v1/threads/{thread_id}`
  - Optional direct thread read for MCP composition.

Read-state routes:

- `POST /v1/messages/read`
  - Request: `agent`, `message_ids`.
  - Idempotently marks messages read.
- `GET /v1/projects/{project_id}/messages/wait`
  - Query: `unread_for`, `timeout_ms`, `limit`, `cursor`.

Notification routes:

- `POST /v1/projects/{project_id}/notifications`
  - Request: `sender`, `content`, optional `task_id`, `metadata`, `urgency`.
  - Creates a message with `intent='notification'`.
- `GET /v1/user-notifications`
  - Cross-project notification feed for user-facing clients.
- `GET /v1/projects/{project_id}/user-notifications`
  - Project-scoped notification feed.
- `POST /v1/user-notifications/read`
  - Request: `agent`, either `notification_ids` or mark-all scope.

Packet routes:

- `POST /v1/projects/{project_id}/tasks/{task_id}/packets/context`
  - Request: role, packet type, requested_by, branch/base/head/scope fields,
    notes, completion reporting mode.
  - May be limited to packet types whose context dependencies are available.
- `GET /v1/projects/{project_id}/tasks/{task_id}/packets/latest`
  - Query: `packet_type`, `role`.
- `GET /v1/projects/{project_id}/packets/{message_id}/worker-prompt`
  - Query: `role`, `completion_reporting_mode`.
- `GET /v1/projects/{project_id}/tasks/{task_id}/completions/latest`
  - Query: `run_id`, `role`.
- `POST /v1/projects/{project_id}/tasks/{task_id}/completions`
  - Storage-only completion packet append. Do not expose as the MCP
    `post_worker_completion_packet` replacement until worker authority handles
    run validation and assignment state.

## Cross-Service Dependencies

Writes:

- Messages calls `projects` `assert-writable` before creating messages,
  notifications, or packet records.
- Messages calls `tasks` read/assert endpoint before task-scoped writes.
- Worker/orchestrator services call messages to append packet records after
  validating their own state.

Reads:

- Tasks and projects may use messages API summaries for compatibility stats;
  do not move message SQL into those services.
- Timeline/observation may read display-only message projections through named
  views if needed, but reads must not drive actuation.
- MCP may compose legacy responses by calling tasks, messages, review, and
  worker services.

Forbidden:

- Messages must not write task rows, project rows, review rows, worker rows,
  agent stream rows, or delivery intent rows.
- Messages must not create delivery intents or wake agent streams from replay.
  Routing services may observe new messages and create attention records, but
  that side effect belongs outside messages and must be explicit.
- Messages must not copy Core worker tables or Core dispatch tables as an
  implementation shortcut.

## Migration And Cutover Plan

1. Create `den_messages` schema and app role.
2. Import or sync `den_core.messages` and `den_core.message_reads` into
   `den_messages`.
3. Preserve message IDs if possible. If IDs cannot be preserved, create an
   import map table and keep MCP/Core routing unchanged until all packet refs
   are translated.
4. Run read-only parity smokes:
   - project task message list;
   - unread filter;
   - thread read;
   - notification feed filters;
   - latest task packet;
   - latest worker completion.
5. Run write smokes against a staging project:
   - send message;
   - reply to thread;
   - mark read;
   - wait for unread message with cursor;
   - send notification and mark explicit/scoped read;
   - append context packet and render worker prompt.
6. Move simple message/notification MCP tools after parity passes.
7. Move packet read tools after packet parity passes.
8. Keep `post_worker_completion_packet` staged until worker-state authority is
   ready.

## Haunt And Regression Smokes

Required tests for the implementation child:

- A replayed or imported message does not create delivery intents, agent-stream
  wakes, or worker assignments.
- `wait_for_messages` returns unread headers and stops with timeout guidance
  instead of encouraging client tight polling.
- Read state is per-agent and idempotent.
- Notification mark-read only marks `intent='notification'` messages.
- Thread replies cannot attach across projects or across mismatched task
  scopes.
- Packet lookup returns the newest matching packet by metadata type/role.
- Worker prompt rendering never embeds the full packet body in the process
  prompt.
- Completion packet readback finds by run/session ID and role without mutating
  worker state.
- Storage-only completion append is not exposed as a full worker completion
  route until worker validation exists.
