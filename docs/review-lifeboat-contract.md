# Review Lifeboat Contract

This is the extraction contract for Den Core review workflow: review rounds,
findings, verdicts, implementer responses, findings packets, and
split-to-follow-up behavior. It follows the projects, tasks, and messages
lifeboat contracts: the review service owns review state and review packet
validation, but it must not become a task lifecycle owner, message store, worker
runtime, or legacy dispatch system.

The successor service is:

- Go package/service: `den-services/review`
- Postgres schema: `den_review`
- Runtime role: `den_review_app`
- Primary tables: `den_review.review_rounds`,
  `den_review.review_findings`, `den_review.review_finding_events`, and
  `den_review.review_packets`
- Project authority: `den-services/projects`, checked through the projects API
  or an approved projects read projection before writes
- Task authority: `den-services/tasks`, checked through the tasks API or an
  approved task projection before writes and packet acceptance
- Message authority: `den-services/messages`, called through API for durable
  task-thread packet records
- Deployment pattern: the lifeboat substrate in
  [`docs/lifeboat-service-substrate.md`](./lifeboat-service-substrate.md)

Do not create a placeholder service only to satisfy a smoke test. Register
`review` in `deployment/services.yaml` only when the real review contract is
implemented.

## Boundary Statement

`review` owns the state machine for review rounds, review findings, finding
statuses, review verdicts, and the deterministic validation of review packet
Markdown. It is the source of truth for whether a finding is open, claimed
fixed, verified fixed, split to follow-up, superseded, or not fixed.

`review` does not own:

- task creation, task status transitions, dependencies, or task history;
- durable message storage, message read state, notification feeds, or worker
  context packet storage;
- worker assignment state, worker launch, wake gates, retries, or completion
  transitions;
- legacy dispatch tables or retired dispatch mutation tools.

Review packets are authored as Markdown with YAML front matter, but workflow
decisions must use typed envelope fields parsed and validated by the service.
The service must never parse free-form prose to decide whether a packet is for
the right task, verdict, reviewed commit, or finding state.

Recommended indexes:

- `den_review.review_rounds(project_id, task_id, round_number)`
- `den_review.review_rounds(project_id, task_id, requested_at desc, id desc)`
- `den_review.review_findings(project_id, task_id, status)`
- `den_review.review_findings(review_round_id, finding_number)`
- `den_review.review_finding_events(review_finding_id, created_at, id)`
- `den_review.review_packets(project_id, task_id, created_at desc, id desc)`
- `den_review.review_packets(review_round_id, packet_kind, created_at desc)`
- `den_review.review_packets(message_id)` where `message_id is not null`

Service rules:

- `project_id` is a scope reference owned by `projects`.
- `task_id` and `follow_up_task_id` are task references owned by `tasks`.
  `review` validates ownership through `tasks` before writes but does not write
  task tables directly.
- Cross-domain cascade deletes are forbidden. If later migrations add foreign
  keys to task/message tables, they must use `ON DELETE RESTRICT` or
  `ON DELETE SET NULL`, not cascade.
- Enum-like values are validated in Go rather than frozen as database `CHECK`
  constraints.
- `review_rounds.project_id` is stored to make review reads and stale-reference
  validation deterministic without joining through task tables.
- `review_finding_events` records status and response history. The current
  columns on `review_findings` remain the latest projection for efficient reads.
- `review_packets` records validation and accepted packet projections. It does
  not replace `messages`; accepted packets are also appended to the task thread
  through the messages API.

- `blocking_bug`
- `acceptance_gap`

- `follow_up_candidate`

Supported finding statuses:

- `open`
- `claimed_fixed`
- `verified_fixed`
- `not_fixed`
- `superseded`
- `split_to_follow_up`

Resolved statuses are `verified_fixed`, `superseded`, and
`split_to_follow_up`. Unresolved statuses are `open`, `claimed_fixed`, and
`not_fixed`.

## Service APIs

Native HTTP routes should use versioned review routes instead of task-owned
review paths:

- `POST /v1/projects/{project_id}/tasks/{task_id}/review/rounds`
  - Creates a round without posting a message packet.
- `GET /v1/projects/{project_id}/tasks/{task_id}/review/rounds`
  - Lists rounds oldest-first.
- `GET /v1/review/rounds/{review_round_id}`
  - Reads one round.
- `POST /v1/review/rounds/{review_round_id}/verdict`
  - Sets verdict and appends the verdict handoff message through messages.
- `POST /v1/review/rounds/{review_round_id}/findings`
  - Creates one structured finding.
- `GET /v1/projects/{project_id}/tasks/{task_id}/review/findings`
  - Lists findings by task with optional `review_round_id`, `status`, and
    `resolved` filters.
- `POST /v1/review/findings/{finding_id}/response`
  - Records implementer response notes and optional status transition.
- `POST /v1/review/findings/{finding_id}/status`
  - Records reviewer/status evidence and optional follow-up link.
- `POST /v1/projects/{project_id}/tasks/{task_id}/review/findings/split-follow-up`
  - Creates a follow-up task through tasks and marks selected findings
    `split_to_follow_up`.
- `GET /v1/projects/{project_id}/tasks/{task_id}/review/workflow-summary`
  - Returns current round, current verdict, unresolved/resolved/addressed
    counts, and review timeline entries.
- `POST /v1/projects/{project_id}/tasks/{task_id}/review/packets/validate`
  - Parses and validates Markdown plus YAML front matter without durable
    acceptance.
- `POST /v1/projects/{project_id}/tasks/{task_id}/review/packets`
  - Validates, persists the typed packet record, applies packet side effects,
    and appends a task-thread message through messages.
- `GET /v1/review/packets/{review_packet_id}`
  - Reads one review packet projection and linked message ID.

## Round Semantics

Creating a review round requires:

- project and task ownership validation through `tasks`;
- task status of `in_progress` or `review`;
- non-empty `requested_by`, `branch`, `base_branch`, `base_commit`, and
  `head_commit`;
- non-negative `commits_since_last_review`, `inherited_commit_count`, and
  `task_local_commit_count` when supplied;
- JSON-array `tests_run` when supplied;
- a new `head_commit` that differs from the latest known reviewed head for the
  same task unless the request explicitly targets the first round.

`round_number` is assigned per `(project_id, task_id)` as latest round plus
one. `last_reviewed_head_commit` defaults to the previous round's
`head_commit`. `delta_base_commit` defaults to `last_reviewed_head_commit` when
available. Preferred diff fields default to the round's base/head branch and
commit. Alternate diff fields are all-or-none enough to identify a real global
diff: if any alternate diff field is supplied, the alternate base ref and base
commit must be valid, and alternate head fields default to the reviewed branch
and head commit.

The preserved round metadata is:

- branch, base branch, base commit, and head commit;
- last reviewed head commit and commits since last review;
- tests run and notes;
- preferred diff base/head refs and commits;
- alternate/global diff base/head refs and commits;
- delta base commit;
- inherited commit count and task-local commit count.

## Finding Semantics

Creating a finding requires:

- a review round that belongs to the supplied project/task;
- a valid category;
- non-empty summary;
- JSON-array file references and test commands when supplied.

`finding_number` is assigned per `(project_id, task_id)` as latest finding plus
one. `finding_key` preserves the Core shape `R{task_id}-{finding_number}` for
human compatibility.

Implementer responses are stored separately from reviewer/status notes:

- `response_by`, `response_notes`, and `response_at` describe the implementer
  answer.
- `status_updated_by`, `status_notes`, and `status_updated_at` describe the
  latest status or verification evidence.

`follow_up_task_id` is accepted only with `split_to_follow_up`. Supplying it
with any other status is a validation error. Setting a non-split status clears
any old follow-up link.

## Split-To-Follow-Up

Split-to-follow-up is a review operation with a tasks side effect:

1. Validate source task ownership and review finding ownership.
2. Reject missing findings and findings outside the source task.
3. Skip `blocking_bug` findings unless `override_blocking=true`.
4. Create one follow-up task through the tasks API with an idempotency key.
5. Mark selected findings `split_to_follow_up` with the returned task ID.
6. Append finding events for each split finding.

The follow-up task description should include:

- source project and task;
- review round number and reviewed branch/head commit;
- finding keys and summaries;
- file references and test commands when present;
- the split actor.

Failure semantics:

- If tasks rejects the follow-up create, no review finding statuses change.
- If follow-up creation succeeds but a later review write fails, return a
  recoverable error that includes `follow_up_task_id` and the idempotency key so
  retry can mark the findings without creating a duplicate task.
- Repeating the same request with the same idempotency key must be idempotent.

## Cross-Service Calls

Review writes use synchronous API calls. Do not create a general event bus or
an executable outbox for review.

Calls to `tasks`:

- validate project/task existence and task status before review packet
  acceptance;
- validate optional parent task ownership for follow-up creation;
- create follow-up tasks for split findings;
- read task status for stale-reference checks.

Calls to `messages`:

- append accepted review request and re-review packets with intent
  `review_request`;
- append review findings and changes-requested packets with intent
  `review_feedback`;
- append looks-good packets with intent `review_approval`;
- append follow-up-needed or blocked-by-dependency handoffs with intent
  `review_feedback`;
- return the accepted `message_id` for storage in `review_packets`.

Message metadata written by review should preserve compatibility keys:

- `schema='den_review_packet'`
- `schema_version=1`
- `type`, such as `review_request_packet`, `rereview_packet`,
  `review_findings_packet`, `review_feedback`, or `merge_request`
- `packet_kind`
- `project_id`
- `task_id`
- `review_round_id`
- `round_number`
- `branch`
- `base_branch`
- `base_commit`
- `head_commit`
- `preferred_diff`
- `alternate_diff`
- `delta_base_commit`
- `verdict` when applicable
- `finding_ids` when applicable
- `run_id` and `subagent_role` when applicable

The review service stores the accepted packet projection before or atomically
with message append according to the implementation's transaction boundary. If
message append fails, the review packet must not be reported as accepted unless
the response clearly marks it `pending_message_append` and exposes a retry-safe
reconciliation path. The first implementation should prefer fail-closed
synchronous acceptance over deferred reconciliation.

## Markdown Packet Contract

Agents submit one Markdown payload with shallow YAML front matter followed by
Markdown narrative. The front matter and extracted typed envelope are
authoritative for routing and workflow decisions. The Markdown body is
canonical for human and reviewer narrative.

Required common front matter:

```yaml
schema: den_review_packet
schema_version: 1
packet_kind: review_request
project_id: den-services
task_id: 3640
sender: pi
```

Supported `packet_kind` values:

- `review_request`
- `rereview_request`
- `review_findings`
- `implementer_response`
- `completion_evidence`

`review_request` and `rereview_request` front matter requires:

```yaml
requested_by: pi
branch: task/3640-review-contract
base_branch: main
base_commit: abc123
head_commit: def456
last_reviewed_head_commit: abc123
commits_since_last_review: 1
preferred_diff:
  base_ref: main
  base_commit: abc123
  head_ref: task/3640-review-contract
  head_commit: def456
delta_base_commit: abc123
tests_run:
  - git diff --check
verify:
  - id: diff_refs_confirmed
    checked: true
  - id: tests_recorded
    checked: true
```

`review_findings` front matter requires:

```yaml
review_round_id: 10
reviewed_head_commit: def456
verdict: changes_requested
findings:
  - key: R3640-1
    category: blocking_bug
    summary: Finding summary
    file_references:
      - docs/review-lifeboat-contract.md:10
    test_commands:
      - git diff --check
verify:
  - id: reviewed_head_matches_round
    checked: true
  - id: findings_have_statusable_categories
    checked: true
```

`implementer_response` front matter requires:

```yaml
review_round_id: 10
reviewed_head_commit: def456
responses:
  - finding_key: R3640-1
    status: claimed_fixed
    summary: Fixed by tightening validation language.
verify:
  - id: finding_refs_checked
    checked: true
```

`completion_evidence` front matter is review-aware but not a worker-state
transition authority. It may include:

```yaml
review_round_id: 10
reviewed_head_commit: def456
verdict: looks_good
finding_ids:
  - 100
tests_run:
  - go test ./...
verify:
  - id: completion_refs_checked
    checked: true
```

The parser should normalize field aliases where compatibility requires it, but
the accepted typed envelope must use one canonical field name. Unknown
front-matter fields are preserved in `front_matter` but ignored for workflow
decisions unless promoted by a future schema version.

Required `VERIFY` items are packet-type-specific. If a required item is absent
or `checked` is not true, validation fails before durable acceptance.

## Validation And Failure Semantics

Packet validation fails closed before durable acceptance when:

- project does not exist or is not writable;
- task does not exist or belongs to another project;
- task status is not reviewable for the packet kind;
- review round does not belong to the supplied project/task;
- finding ID or key does not belong to the supplied project/task/round;
- verdict, category, or status enum is invalid;
- reviewed head commit is missing where required;
- reviewed head commit does not match the referenced review round;
- a same-head review request repeats the last reviewed head without an explicit
  re-review reason;
- a packet references a `done` or `cancelled` task for mutable review state;
- a required `VERIFY` item is absent or unchecked;
- front matter is malformed, not YAML, or not a mapping;
- Markdown body is empty when the packet kind requires narrative evidence.

Reviewable task states:

- `review_request` and `rereview_request`: `in_progress` or `review`
- `review_findings`: `review`
- `implementer_response`: `in_progress` or `review`
- `completion_evidence`: `review`

Structured error responses should use this shape:

```json
{
  "error": "validation_failed",
  "code": "invalid_verdict",
  "field": "verdict",
  "message": "verdict must be one of changes_requested, looks_good, follow_up_needed, blocked_by_dependency.",
  "docs_ref": "tool_docs(\"post_review_packet_markdown\", \"review_findings.verdict\")"
}
```

Validation errors should point to focused tool-documentation lookups, not broad
Den document searches. Suggested docs references:

- `tool_docs("post_review_packet_markdown", "common.project_id")`
- `tool_docs("post_review_packet_markdown", "review_request.head_commit")`
- `tool_docs("post_review_packet_markdown", "review_findings.verdict")`
- `tool_docs("post_review_packet_markdown", "implementer_response.responses")`
- `tool_docs("post_review_packet_markdown", "verify.required_items")`

## Workflow Summary

`GET /v1/projects/{project_id}/tasks/{task_id}/review/workflow-summary`
returns enough data for `get_task_workflow_summary`, reviewer packet builders,
and orchestrator state evaluation without letting `tasks` own review SQL:

- current round;
- current verdict;
- review round count;
- unresolved finding count;
- resolved finding count;
- addressed finding count;
- open findings;
- resolved findings;
- timeline entries with round number, branch, requested by/at, head commit,
  last reviewed head, commits since last review, verdict by/at, total findings,
  open findings, addressed findings, claimed fixed findings, and resolved
  findings.

Reviewer context packets may include this projection through MCP composition or
messages packet builders. Worker/orchestrator services must still own
executable decisions and state transitions.

## Migration And Cutover

1. Create `den_review` schema and service migrations.
2. Import Core `review_rounds` and `review_findings` preserving IDs, round
   numbers, finding numbers, finding keys, and timestamps.
3. Backfill `project_id` by reading tasks before cutover.
4. Backfill `review_finding_events` from latest response/status fields as
   import events where detailed history is absent.
5. Keep Core/MCP routing unchanged until read parity passes for rounds,
   findings, verdicts, and workflow summary.
6. Route low-level review reads and writes to `review`.
7. Route packet validation and packet posting after messages append parity is
   available.
8. Route split-to-follow-up after tasks create API idempotency is available.
9. Keep old dispatch mutation paths retired; do not recreate them in review.

## Smoke Tests

Required service smokes:

- Create a first review round with branch/base/head metadata and read it back.
- Create a re-review round and verify last reviewed head and delta defaults.
- Reject a same-head review request without explicit re-review context.
- Create findings in all categories and verify task-scoped finding numbers and
  `R{task_id}-{finding_number}` keys.
- List findings by task, by round, by status, and by resolved filter.
- Record an implementer response and verify response fields do not overwrite
  status evidence.
- Set finding status to each supported value and reject invalid enum values.
- Reject `follow_up_task_id` unless status is `split_to_follow_up`.
- Set verdicts and verify the correct message intent and metadata are appended.
- Split non-blocking findings to a follow-up task and verify idempotent retry.
- Verify blocking findings are skipped unless `override_blocking=true`.
- Validate a good Markdown review request packet without persisting it.
- Post a good Markdown review findings packet and verify `review_packets` plus
  the task-thread message evidence.
- Reject malformed front matter, wrong project/task, stale reviewed head,
  invalid verdict/status, and unchecked required `VERIFY` items.
- Return workflow summary counts and timeline entries matching the imported or
  newly written review state.
- Verify no route or tool writes old legacy dispatch state.

## Non-Goals

- Do not implement worker assignment completion, retry, wake, or gate logic in
  review.
- Do not store durable messages in review beyond the accepted packet projection
  and linked `message_id`.
- Do not move `get_task` or `get_task_workflow_summary` wholesale into tasks.
- Do not require agents to fill a large parameter list when the Markdown packet
  path can validate a typed envelope.
- Do not parse architectural alignment or code correctness from prose. Reviewers
  read the Markdown, then spot-check load-bearing claims in code and artifacts.
