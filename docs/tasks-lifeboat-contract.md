# Tasks Lifeboat Contract

This is the extraction contract for Den task lifecycle, dependencies, subtasks,
availability, and task history. It follows the projects/scope contract: the
tasks service owns task writes and task-local invariants, but it does not become
a new Core-shaped aggregator for messages, review, worker runs, or documents.

The successor service is:

- Go package/service: `den-services/tasks`
- Postgres schema: `den_tasks`
- Runtime role: `den_tasks_app`
- Primary tables: `den_tasks.tasks`, `den_tasks.task_dependencies`,
  `den_tasks.task_history`
- Project authority: `den-services/projects`, read through API or approved
  read projection, with no raw cross-schema writes
- Deployment pattern: the lifeboat substrate in
  [`docs/lifeboat-service-substrate.md`](./lifeboat-service-substrate.md)

Do not create a placeholder service only to satisfy a smoke test. Register
`tasks` in `deployment/services.yaml` only when the real contract is
implemented.

## Existing Core Inventory

Core task ownership is currently centered in:

- `TaskRepository`
- `ProjectTask`
- `TaskHistoryEntry`
- `TaskRoutes`
- `TaskTools`
- blocked-task escalation services tied to blocked status transitions

Current task tables:

```sql
create table den_core.tasks (
    id bigint generated by default as identity primary key,
    project_id text not null references den_core.projects(id) on delete cascade,
    parent_id bigint references den_core.tasks(id) on delete cascade,
    title text not null,
    description text,
    status text not null default 'planned',
    priority integer not null default 3,
    assigned_to text,
    tags jsonb,
    blocker_summary text,
    blocker_reason text,
    blocker_attempted_remedies text,
    blocker_suggested_next_step text,
    blocker_requires_human_input boolean not null default false,
    created_at timestamptz not null default now(),
    updated_at timestamptz not null default now()
);

create table den_core.task_dependencies (
    task_id bigint not null references den_core.tasks(id) on delete cascade,
    depends_on bigint not null references den_core.tasks(id) on delete cascade,
    primary key (task_id, depends_on),
    check (task_id <> depends_on)
);

create table den_core.task_history (
    id bigint generated by default as identity primary key,
    task_id bigint not null references den_core.tasks(id) on delete cascade,
    field text not null,
    old_value text,
    new_value text,
    changed_by text,
    changed_at timestamptz not null default now()
);
```

Current task status values are `planned`, `in_progress`, `review`, `blocked`,
`done`, and `cancelled`. The successor service should validate status and
priority in Go rather than freezing status as a database `CHECK` constraint.
Priority remains an integer from 1 through 5.

Current REST behavior to preserve:

- `POST /api/projects/{projectId}/tasks/` creates a task or subtask and may add
  dependencies.
- `GET /api/projects/{projectId}/tasks/` lists task summaries, top-level only
  by default.
- `GET /api/projects/{projectId}/tasks/{taskId}` returns detail with
  dependencies, subtasks, recent messages, and review state.
- `PUT /api/projects/{projectId}/tasks/{taskId}` applies non-null task field
  updates, records history, and triggers status side effects.
- `GET /api/projects/{projectId}/tasks/next` returns the next unblocked task.
- `POST /api/projects/{projectId}/tasks/{taskId}/dependencies` adds a
  dependency and rejects cycles.
- `DELETE /api/projects/{projectId}/tasks/{taskId}/dependencies/{dependsOn}`
  removes a dependency.

Current task MCP tools:

- `create_task`
- `list_tasks`
- `get_task`
- `get_task_workflow_summary`
- `update_task`
- `next_task`
- `add_dependency`
- `remove_dependency`

Review tools currently live beside task routes in Core, but review remains a
separate later lifeboat domain. This contract only owns task lifecycle,
dependency graph, subtasks, availability, and task-local history.

## Inbound Dependencies

Known task dependents include:

- messages, notifications, task-thread packets, latest-packet lookup, and
  prompt rendering;
- review rounds, review findings, verdict workflow, finding split-to-follow-up,
  and review workflow summaries;
- worker assignments, worker runs, completion packets, no-capacity records, and
  orchestrator leases;
- agent stream entries and active/drain projections;
- capabilities and usage events with caller task IDs;
- collaboration sessions, desktop snapshots, workspace/git state, and
  artifacts;
- librarian and context packet builders that need task summaries, dependencies,
  messages, and review state.

Dependent domains should store `task_id bigint` as a domain reference. They must
not write `den_tasks` tables directly. Cross-domain cascade deletes are
forbidden outside the tasks schema; if a later schema uses a foreign key to
`den_tasks.tasks`, it must be `ON DELETE RESTRICT` or `ON DELETE SET NULL`, not
cascade.

## New Schema Boundary

The owning schema is `den_tasks`.

```sql
create table den_tasks.tasks (
    id bigint generated by default as identity primary key,
    project_id text not null,
    parent_id bigint references den_tasks.tasks(id) on delete cascade,
    title text not null,
    description text,
    status text not null default 'planned',
    priority integer not null default 3,
    assigned_to text,
    tags jsonb,
    blocker_summary text,
    blocker_reason text,
    blocker_attempted_remedies text,
    blocker_suggested_next_step text,
    blocker_requires_human_input boolean not null default false,
    created_at timestamptz not null default now(),
    updated_at timestamptz not null default now()
);

create table den_tasks.task_dependencies (
    task_id bigint not null references den_tasks.tasks(id) on delete cascade,
    depends_on bigint not null references den_tasks.tasks(id) on delete cascade,
    primary key (task_id, depends_on),
    check (task_id <> depends_on)
);

create table den_tasks.task_history (
    id bigint generated by default as identity primary key,
    task_id bigint not null references den_tasks.tasks(id) on delete cascade,
    field text not null,
    old_value text,
    new_value text,
    changed_by text,
    changed_at timestamptz not null default now()
);
```

Recommended indexes:

- `den_tasks.tasks(project_id, status)`
- `den_tasks.tasks(project_id, parent_id, priority, id)`
- `den_tasks.tasks(assigned_to)` where `assigned_to is not null`
- `den_tasks.tasks(parent_id)` where `parent_id is not null`
- `den_tasks.task_dependencies(depends_on)`
- `den_tasks.task_history(task_id, changed_at desc, id desc)`

Service rules:

- `project_id` is a scope reference owned by the projects service. The tasks
  service validates project existence and archive write policy through the
  projects API or an approved projects read projection before creates and
  non-archival updates.
- `parent_id` must reference a task in the same project.
- A task cannot be its own parent.
- The parent chain must not contain cycles.
- Dependency edges must stay within a project.
- A task cannot depend on itself.
- Adding a dependency must reject direct and transitive cycles.
- `done`, `cancelled`, and `review` dependencies do not block availability.
  `review` means the dependency has a stable reviewable handoff point and can
  unblock safe downstream work while review runs in parallel; it is not final
  approval.
- `tags` are JSON arrays of strings. Tag filters use AND semantics and must not
  match substrings.
- `task_history` records every changed mutable field with old/new values and
  `changed_by`.

## Availability

Availability is a projection, not an independently mutable column:

- `planned` with no unfinished dependencies: `available`
- `planned` with one or more unfinished dependencies: `waiting_on_dependencies`
- `in_progress`: `in_progress`
- `review`: `review`
- `blocked`: `blocked`
- `done`: `done`
- `cancelled`: `cancelled`

Unfinished dependencies are dependencies whose status is not `done`, not
`cancelled`, and not `review`. Dependencies in `review` are treated as
provisionally satisfied for scheduling so agents can continue with downstream
work while review runs in parallel.

`next_task` ordering must preserve Core behavior:

1. Candidate tier 1: `planned` or `in_progress` subtasks whose parent is
   `in_progress` or `review`.
2. Candidate tier 2: top-level `planned` tasks.
3. Exclude tasks with unfinished dependencies. A dependency in `review` is not
   unfinished for scheduling, but downstream work should treat it as provisional
   until the dependency is approved or done.
4. Optional `assigned_to` filter applies in both tiers. When present, it matches
   tasks assigned to that agent **and** unassigned tasks, so unassigned work can
   be picked up by any capable worker.
5. Order by tier, priority ascending, dependency count ascending, then ID
   ascending.

## Blocked-State Invariants

Transitioning into `blocked` from another status requires structured blocker
context:

- `blocker_summary`
- `blocker_reason`

Optional fields:

- `blocker_attempted_remedies`
- `blocker_suggested_next_step`
- `blocker_requires_human_input`, default `false`

The tasks service owns validation and persistence of these blocker fields.
Notifications, planner wakeups, or escalation delivery are side effects owned by
messages/agent-stream/delivery successor services. During migration, MCP may
keep blocked escalation composed in Core or a compatibility shim until those
services exist.

## Service API

Native routes should live under `/v1`.

Task routes:

- `POST /v1/projects/{project_id}/tasks`
  - Request: `title`, optional `description`, `priority`, `tags`,
    `assigned_to`, `depends_on`, `parent_id`.
  - Validates project scope and archived-project write policy.
- `GET /v1/projects/{project_id}/tasks`
  - Filters: `status`, `assigned_to`, `tags`, `priority`, `parent_id`,
    `tree=false`.
  - Default list is top-level tasks only.
  - `tree=true` with no `parent_id` includes all project tasks.
- `GET /v1/tasks/{task_id}`
  - Metadata-only task read plus dependencies, subtasks, and history window.
- `PATCH /v1/tasks/{task_id}`
  - Non-null patch semantics for `title`, `description`, `status`, `priority`,
    `assigned_to`, `tags`, `parent_id`, and blocker fields.
  - Requires `agent` or equivalent actor identity for audit history.
- `GET /v1/projects/{project_id}/tasks/next?assigned_to=`
  - Returns the next unblocked task or an empty/no-task response.
- `POST /v1/tasks/{task_id}/dependencies`
  - Request: `depends_on`.
- `DELETE /v1/tasks/{task_id}/dependencies/{depends_on}`
  - Idempotent remove is acceptable.
- `GET /v1/tasks/{task_id}/history`
  - Returns task-local history entries, newest first by default.

Project-scoped aliases may also be exposed for compatibility:

- `GET /v1/projects/{project_id}/tasks/{task_id}`
- `PATCH /v1/projects/{project_id}/tasks/{task_id}`
- `POST /v1/projects/{project_id}/tasks/{task_id}/dependencies`
- `DELETE /v1/projects/{project_id}/tasks/{task_id}/dependencies/{depends_on}`

The service must reject project/task mismatches on project-scoped aliases.

## Detail and Workflow Composition

Core `get_task` includes messages and review data, and legacy
`get_task_workflow_summary` did the same before MCP composition. Tasks service
must not move message or review SQL into `tasks`.

Compatibility should be staged:

1. Tasks service owns canonical task metadata, dependencies, subtasks,
   availability, and history.
2. MCP may route `create_task`, `list_tasks`, `update_task`, `next_task`,
   `add_dependency`, and `remove_dependency` to tasks after import/parity
   smokes pass.
3. MCP keeps `get_task` Core-routed, or MCP-composed, until messages and
   review successor APIs expose the required summaries.
   `get_task_workflow_summary` is MCP-composed once tasks, messages, and review
   successor APIs exist.
4. Once source services exist, MCP composes legacy detail from tasks metadata,
   messages headers, and review summaries.

This keeps tasks as a lifecycle authority rather than an aggregation hub.

## Parity Smokes

Run against a migrated staging copy before flipping MCP routes:

1. Snapshot Core and successor `list_tasks` for a project with default filters.
   Top-level tasks, dependency counts, unfinished dependency counts, subtask
   counts, tags, and availability must match.
2. Compare `list_tasks` filters for:
   - `status=planned,in_progress`;
   - `assigned_to`;
   - `tags` with AND semantics;
   - `priority`;
   - `parent_id`;
   - `tree=true`.
3. Create a task with tags, assignment, priority, and description; read it back.
4. Create a subtask and verify parent/subtask list behavior.
5. Add a dependency; verify dependency readback, `waiting_on_dependencies`, and
   `next_task` exclusion.
6. Move the dependency to `review`; verify availability becomes `available` and
   `next_task` can select the dependent task while the dependency is pending
   review.
7. Complete or cancel the dependency; verify availability remains `available`
   and `next_task` can select the task.
8. Attempt direct and transitive dependency cycles; both must fail.
9. Update task fields and verify history readback records field, old value, new
   value, actor, and timestamp.
10. Transition to `blocked` without blocker fields; verify rejection.
11. Transition to `blocked` with structured blocker fields; verify persistence
    and `blocked` availability.
12. Verify project-scope mismatch rejection for project-scoped aliases.

## Cutover Sequence

1. Implement `den_tasks` schema, service, and tests.
2. Import/sync `den_core.tasks`, `task_dependencies`, and `task_history`.
3. Run read-only parity smokes without MCP route changes.
4. Route write/read task lifecycle tools that do not need messages/review to
   `tasks`.
5. Keep `get_task` staged until message/review successor summaries are
   available; compose `get_task_workflow_summary` in MCP once those source
   APIs exist.
6. Remove any temporary dual-write/import mode quickly; avoid indefinite dual
   writers.

The goal is a clean task lifecycle service, not a hidden replacement for all
task-adjacent Core behavior.
